# Repository Audit - 2026-05-07

## Summary

The repository has been cleaned and brought back to a source-first state. Generated datasets, result files, model binaries, scratch scripts, logs, and local databases are no longer tracked. Documentation now reflects the current CLI behavior, artifact policy, and model workflow.

## Key Findings

- The previous `bounty_stats.db` leaderboard was not trustworthy. Most rows were vibe-only records with blank title/repo metadata and null amounts.
- Null amounts did not prove the parser was globally broken. The primary problem was that vibe-only rows could be inserted with production-looking defaults.
- The parser and tests had drifted: scripts expected `extract_bounty_amount()`, while the module exposed `parse_numeric_amount()`.
- Documented scrape commands placed options after `scrape`, but argparse originally accepted several of those options only before the subcommand.
- Root-level generated artifacts made the repository look less reliable than the code actually was.

## Fixes Completed

- Restored `extract_bounty_amount()` as a compatibility wrapper.
- Standardized unknown bounty amounts to `-1.0` when bounty cues exist but no amount is parseable.
- Made `detect_snipe()` compatible with both pipeline calls and timeline-only helper calls.
- Allowed scrape runtime options before or after the `scrape` subcommand.
- Mapped compatibility CLI fields into runtime config fields.
- Marked new vibe-only rows as `lead_mode='vibe_only'`, `escrow_verified=0`, and `score=0`.
- Excluded incomplete vibe-only rows from lead inspection and dataset export.
- Added regression tests for CLI behavior, DB row filtering, and model feature metadata.
- Removed generated datasets, model binaries, result JSON, and stale sidecar files from source control.
- Rewrote `.gitignore`, `README.md`, `CONTRIBUTING.md`, and `docs/walkthrough.md`.
